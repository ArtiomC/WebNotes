UNDOING CHANGES

To undo changes we need to know on which level (на каком уровне) we have the changes we want to undo are located.

Working directory (рабочая дериктория в которой уже есть измененные файлы)

If we are in the working directory and files are not added to the staging area (also called index), we are able to use the below commands.

While the changes or a file are in working directory, the below commands are allowed to be used.

  • git checkout --file.txt
  • git checkout .
  • git clean -xdf

We use the ls command to see the files that we have on the local branch. We check the status of the repo with the git status command. The working tree is clean, there is nothing to commit.

After that we have updated the text in documents using text editor. After the changes were saved in the text editor, the status of the local git repo has changed. The existing files have a modified status. 

If we had only a few documents where we would like to undo the changes, we could do it manually. However if there is a need to undo changes in fifty documents, doing it manually may be really time consuming.

So now we have two modified files and we want to get things back as they were.

We can discard the changes for a single file as it’s suggested in the terminal.

After the changes were discarded for the README.md file, only the song.txt file is in modified status. When We checked the text editor, there was a request to reload the page since there were changes made to the file.

And how do we discard changes for multiple files at the same time? There is a command for that restore .

The . means - "Restore all files from the current directory and lower. (Сделать restore всех файлов от текущей дериктории и ниже)". Since we are in the root on the repository, GIT will restore all the files to the previous state ewhich is known to git. (Так как мы находимся в корне репозитория он вернет исходное состояние всех файлов известнах гиту)

Basically the git restore <file> and restore . commands will undo the chages up until the version that GIT is familiar with. That only works with preexisting files which were modified and there is a version toward which the changes may be "undone".

Important

If you do a checkout . (or restore .) all of the changes in the files affected with the command will be lost permanently. Be careful with that. If you worked on somethong awesome all day and then used the restore . command - all of your changes are lost permanently. Almost all actions in GIT are final.

Look up the difference between checkout . and restore . commands.

What will happen if we have a couple of new files? Let's create two new files which don't have a version stored in the repository, since they were just created.

The restore command will have no effect on the new files sonce they have no history in GIT yet and GIT will not remember their previous state. That's why we can't roll back the files to the previous verion, cause there is no previous version.

That's when the git clean -xdf command comes in handy.

The git clean command has a lot of flags. Three of them were used in the above example -xdf. Let’s see what each of the flags stands for.

  • x – it means we are potentially ignoring rules of the .gitignore file.
  • d – it means we will delete also directories if they are not known to git ( означает, что я удаляю в том числе и директории если они не известны гиту)
  • f – stands for “force”. If the f clag is not called upon, the command will not go through.
